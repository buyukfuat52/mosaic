# ACCEPTANCE CRITERIA

## Functional requirements

- [ ] YOLO annotation rows are parsed correctly.
- [ ] Empty rows are handled safely.
- [ ] Malformed rows are reported.
- [ ] YOLO normalized coordinates convert to pixel XYXY.
- [ ] Pixel XYXY coordinates convert back to YOLO normalized format.
- [ ] 2x2 mosaics are generated using 4 cells.
- [ ] 3x3 mosaics are generated using 9 cells.
- [ ] Letterbox scale and padding are used in bounding-box transforms.
- [ ] Cell offsets are included in transformed coordinates.
- [ ] Bounding boxes are clipped to the mosaic boundary.
- [ ] Reversed coordinates are repaired.
- [ ] Non-finite values are rejected.
- [ ] Zero-area boxes are rejected.
- [ ] Minimum visible-area filtering is applied.
- [ ] Output annotations are written in valid YOLO format.
- [ ] Bounding-box preview images can be generated.
- [ ] Synthetic datasets can be generated.
- [ ] CLI exposes `generate`, `validate`, `visualize`, `synthetic`, `benchmark`, and `web`.
- [ ] Gradio can generate mosaics.
- [ ] CLI and web UI share the same service layer.
- [ ] Web workflow can export outputs as a ZIP archive.

## Testing requirements

- [ ] Geometry tests use manually verifiable expected values.
- [ ] The 2x2 bottom-right example produces `(0.75, 0.75, 0.5, 0.5)`.
- [ ] 3x3 tests cover at least one center cell and one corner cell.
- [ ] Deterministic seed behavior is tested.
- [ ] All missing-label policies are tested.
- [ ] CLI smoke tests exist.
- [ ] Synthetic data generation is tested.
- [ ] Malformed rows are tested.
- [ ] Visualization dimensions are tested.
- [ ] Service-layer integration is tested.
- [ ] All tests pass.
- [ ] Total coverage is at least 90%.
- [ ] Geometry and validation branches receive focused coverage.
